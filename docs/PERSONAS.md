# korea-linkhub — Personas

10명의 외국인 페르소나로 한국 방문 케이스 전반을 커버한다.

**Type A (시간 축)** — 같은 사람이 거치는 단계
**Type B (목적/상황 축)** — 단계와 독립적인 케이스

새 데이터/UX 결정 시 "어떤 페르소나의 어떤 순간을 돕는가?"를 먼저 답한다. 어느 페르소나에도 속하지 않는 항목은 추가하지 않는다.

## Type C — Internal Evaluator (제품의 가상 평가자, 사용자 아님)

### 11. Rene — Senior Design Critic
- **DE 71, 산업 디자인 30년 + 디지털 UI 15년**, Dieter Rams 정신적 후예
- 언어: de(L1), en(상)
- **역할:** 사이트의 시각·정보 디자인 결함을 **측정 가능한 지표**로 평가한다. 사용자가 아니라 제품의 내부 평가자.
- **평가 기준:** Rams 10원칙 + Norman 사용성 휴리스틱 8개 + Apple HIG 모바일 가이드 = 총 16개
- **성향:** 칭찬 ≤ 비판 1:3. 추상 표현("더 깔끔하게") 거부. 항상 픽셀·간격·컬러·타이포 단위로.
- **운영:** 에이전트 `critic` (`.claude/agents/critic.md`) + 스킬 `/linkhub-design-review`
- **출력:** PASS/WARN/FAIL 점수표 + 처방(Tailwind 클래스 단위) + 위임 카드(ux/frontend/curator)

> **Rene는 페르소나라기보다 "제품 평가관"이다.** 다른 10명이 "사이트를 사용하는 사람"이라면 Rene는 "사이트를 평가하는 사람". 그가 통과시킨 디자인만 사용자(10명) 앞에 나간다.

## 페르소나 사용 가이드

### 신규 데이터 추가 시 (curator)
1. 이 항목은 누구를 돕는가? (페르소나 1명 이상 지정)
2. 매트릭스에서 그 페르소나의 카테고리 값을 priority 시작점으로
3. 겹치는 페르소나 ≥3명이면 +10 가산

예) "1330 관광공사 통역안내"
- Maya(100) + Hassan(95) + Lucas(80) + Yuki(90) + Diego(95) + Fatima(90) = 6명
- priority = max(100) + 보너스 = **100**

### UX 변경 시 (ux + frontend)
1. Hassan의 3탭/5초 SOS 도달 깨면 reject
2. priority ≥95 항목은 검색 0건일 때도 fallback 노출
3. Aiko 사전 계획 시나리오에서 PC와 모바일 모두 작동
4. Yuki(10대 첫 해외) 가독성 — 작은 폰트/약한 contrast 금지
5. Min-jun(한국어 전혀 모름) 화면 — 한자/한글 의존 X

### 다국어 추가 시 (translator)
1. 페르소나 모국어부터 — vi(Linh), pt-BR(Lucas), es(Carlos+Diego), ar 데이터만(Hassan+Fatima)
